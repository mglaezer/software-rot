# Software Rot in the Age of AI: An Evidence-Based Analysis (2025–2026)

**Technical Analysis Report**

**March 15, 2026**

## Abstract

This report synthesizes empirical evidence on software rot — the progressive decay of software systems — and examines how AI-assisted development changes its dynamics. Drawing on peer-reviewed studies, large-scale industry analyses, and established research programs, it makes the case that software architecture and design discipline are more critical than ever when code is generated by AI.

**Scope note:** This report focuses on software rot — structural decay, maintainability, and technical debt. Security vulnerabilities in AI-generated code are a significant and well-documented concern, but are outside the scope of this analysis.

---

## 1 The Core Problem: What Is Software Rot?

**Software Rot** (or "Bit Rot") refers to the functional decay of a system over time, even if the code itself remains unchanged. Unlike physical materials, software does not wear out; rather, it rots because its **internal environment** becomes increasingly rigid while the **external environment** (OS, APIs, dependencies, standards) continues to evolve.

This phenomenon is well-established in software engineering. Lehman's Laws of Software Evolution, empirically validated across open-source projects including Apache Tomcat and Apache Ant, confirm that evolving software systems exhibit both continuing growth in size and increasing complexity over time [1].

### 1.1 Why Rot Accelerates

The acceleration has a structural explanation. In any system with N components, the number of potential interdependencies grows as N(N − 1)/2. As features are added linearly, the complexity of managing interactions increases at a faster rate. This creates a "ripple effect" where a single change in a tightly coupled system triggers cascading issues across dependent components.

This is not merely theoretical. An empirical study of 207 open-source Java projects found that projects with minimal refactoring ("Steady Inactive" pattern) show declining code quality metrics over time, with increasing architectural problems as complexity accumulates unchecked [2]. The acceleration is real and measurable.

## 2 The Scale of the Problem

### 2.1 Global Fragility (CAST 2025)

The **2025 "Coding in the Red" Report** by CAST analyzed over 10 billion lines of code across 17 countries representing 51% of global GDP [3]:

- 45% of the world's code is deemed **fragile**, susceptible to failure when facing unexpected conditions [3].
- 32% of code suffers from **bloat**, driving up compute and energy costs [3].
- 31% is too **rigid**, slowing businesses' ability to make changes [3].
- It would take the world's 25 million developers **9 years** of full-time work just to pay off current global technical debt [3].

### 2.2 The Business Impact (McKinsey 2025)

McKinsey's **"Tech Debt: Reclaiming Tech Equity"** analysis of 220 companies across 5 geographies and 7 sectors quantified the business consequences [4]:

- Tech debt accounts for **20 to 40 percent** of the value of companies' entire technology estate before depreciation.
- **10 to 20 percent** of the technology budget for new products is diverted to resolving debt-related issues.
- Companies in the top quintile for Technical Debt Score see revenue growth **20% higher** than those in the bottom quintile.

In the US alone, CISQ estimated the total cost of poor software quality at **$2.41 trillion**, with $1.52 trillion in accumulated technical debt [5]. Gartner reports that organizations that effectively manage technical debt achieve at least **50% faster** service delivery times compared to those that do not [6].

## 3 AI Accelerates Software Rot

The argument that LLMs make architecture unnecessary faces significant empirical evidence to the contrary. AI doesn't eliminate rot; it **compresses the timeline**.

### 3.1 Doubled Code Churn, Declining Refactoring

GitClear's analysis of 211 million changed lines of code (2020–2024) reveals a clear pattern [7][8]:

- **Code churn** — the proportion of new code reverted within two weeks — has **doubled** since the widespread adoption of AI coding tools.
- **Refactoring declined** from 25% of changed lines in 2021 to less than 10% in 2024.
- **Code duplication** (copy-pasted lines) rose from 8.3% to 12.3%, a 48% relative increase.

AI tools can suggest valid code but struggle to reuse and modify existing code — generating new code rather than refactoring what exists [8]. This is a recipe for accelerated rot.

### 3.2 The Compounding Cost

Analysis of AI-generated software costs identifies an inflection point at **16–18 months**, after which maintenance costs for unmanaged AI-generated code can reach **four times traditional levels** by the second year [9]. Roughly 20% of package dependencies suggested by AI do not exist in official repositories ("package hallucination"), introducing fragility from Day 1 [9].

An empirical study of 477 repositories found that LLM-related codebases remain debt-free 2.4x longer than traditional ML repositories, but then accumulate debt rapidly — and once introduced, **only 4.2% of that debt is ever removed** [10].

### 3.3 The Verification Tax

**64% of development teams** report that verifying AI-generated code takes as long as, or longer than, writing the code from scratch [11]. Without architectural guidance, this verification burden becomes unmanageable as a project grows.

A randomized controlled trial by METR found that experienced open-source developers actually took **19% longer** to complete tasks when using AI tools in mature codebases — despite expecting a 24% speedup [12]. The gap between perceived and actual productivity suggests the verification overhead is systematically underestimated.

### 3.4 AI Amplifies What Already Exists

Google's DORA program — the largest ongoing study of software delivery performance — found that AI adoption has a positive relationship with throughput but continues to have a **negative relationship with software delivery stability** [13]. AI acts as an amplifier: it makes existing good practices more effective and existing bad practices more destructive. The greatest return comes not from AI tools themselves but from quality of internal platforms, clarity of workflows, and team alignment.

An analysis of 470 real-world open-source pull requests found that AI-generated PRs average **1.7x more issues** per PR than human-written PRs, with **2.25x more logic and business-logic errors** [14].

### 3.5 AI Cannot Reason Architecturally

An empirical study testing AI agents generating microservices found that current agents are general-purpose code generators **lacking architectural reasoning** for system-level concerns. Regenerating one microservice caused authentication failures across dependent services and inconsistent field names that broke frontend integration tests [15].

A separate study introduced the concept of **"Hallucinated Coupling"** — when an LLM correctly implements a class but incorrectly imports dependencies, violating inversion of control and creating structural debt invisible at the function level [16].

These findings explain why AI-generated code can appear correct in isolation while accelerating systemic rot.

## 4 Architecture Is the Antidote

### 4.1 Refactoring Patterns That Work

The empirical study of 207 open-source Java projects identified four release-wise refactoring patterns [2]. The **"Late Active" pattern** — where teams significantly increase refactoring density as they approach a release — leads to the best code quality outcomes. This pattern prevents complexity from compounding unchecked by systematically addressing architectural drift before each release.

### 4.2 Architecture at Scale: Meta's Approach

Meta's WhatsApp engineering team published their experience deploying AI code generation at enterprise scale over 25 months, producing 3,000+ accepted code changes [17]. Their key finding: **architectural guardrails are essential**. Their system uses cross-session memory, explicit rules, and retrieval-augmented generation to preserve architectural coherence and enforce engineering standards. Without these constraints, AI-generated code drifted from organizational patterns.

### 4.3 What Researchers and Practitioners Say

The consensus among leading software engineering voices is clear:

- **Martin Fowler** (Chief Scientist, Thoughtworks): AI refactoring only delivered functionally correct results **37% of the time**. Refactoring discipline is more important than ever. His definition of architecture — "those decisions which are both important and hard to change" — becomes even more critical when AI generates code rapidly [18].

- **Kent Beck** (creator of Extreme Programming, TDD pioneer): TDD is a "superpower" when working with AI agents — unit tests prevent AI from introducing regressions. Reports that AI agents sometimes delete tests to make them "pass." Architecture and design discipline are the guardrails that make AI productive rather than destructive [19].

- **Dave Farley** (co-author of *Continuous Delivery*): Called vibe coding "2025's worst idea." Teams practicing continuous delivery see dramatic improvements because they naturally work incrementally with feedback — a natural fit for AI-assisted development. Architecture and engineering discipline are non-negotiable prerequisites [20].

### 4.4 The Gartner Warning

Gartner predicts that by 2028, prompt-to-app approaches by citizen developers will increase software defects by **2500%**, triggering a software quality and reliability crisis. They recommend governance, validation frameworks, and architectural controls to capture productivity gains while avoiding the quality crisis [21].

## 6 Summary of Findings

- **Software rot is real and costly.** 45% of the world's code is fragile [3], and tech debt accounts for 20–40% of technology estate value [4]. Lehman's Laws confirm that complexity increases continuously in evolving systems [1].
- **AI accelerates rot.** Code churn has doubled [7], refactoring has declined by 60% [8], and AI-generated PRs contain 1.7x more issues [14].
- **AI cannot replace architecture.** Current AI tools lack architectural reasoning [15][16] and produce "Hallucinated Coupling" that violates design principles invisibly.
- **Architecture amplifies AI's benefits.** DORA research shows AI makes good practices more effective [13]. Meta's enterprise deployment proves architectural guardrails are essential at scale [17].
- **Refactoring discipline is the key.** The Late Active refactoring pattern is the most effective way to manage complexity [2], and leading practitioners agree that TDD and design discipline are more important than ever [18][19][20].
